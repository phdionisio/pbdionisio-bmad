# Decision Log — Ledger & Processing Cash Advances and Liquidation PRD

**Status:** Draft

---

## DL-001 — Two-layer architecture

**Date:** 2026-05-22
**Decision:** The system is split into two distinct layers: a generic Ledger library (`pbdionisio-ledger` in `libs/`) and an application-layer Processing tool in `apps/`. The Ledger has no knowledge of BRP workflows or application types.
**Rationale:** User confirmed the Ledger will be reused by future tools. Keeping it generic prevents tight coupling and preserves reusability.
**Captured in PRD:** §5 Architecture Overview, §6 FR-L1–FR-L5

---

## DL-002 — Void instead of delete

**Date:** 2026-05-22
**Decision:** Transactions are never deleted. The void operation marks a transaction as cancelled with an immutable reason. Voided records are retained indefinitely.
**Rationale:** Preserves audit trail. A deleted record would leave a gap in the money trail.
**Role constraint:** Void requires Manager or above. Reason is mandatory.
**Captured in PRD:** §4 User Roles, §6 FR-L2

---

## DL-003 — Hard balance enforcement at Ledger level

**Date:** 2026-05-22
**Decision:** The Ledger enforces that no debit transaction can be approved or settled if it would cause the fund balance to go negative. This constraint cannot be bypassed by any role.
**Rationale:** User said "enforce it" when asked whether balance limits should be advisory or hard. Preventing overdrafts is the core integrity guarantee of the Ledger.
**Captured in PRD:** §6 FR-L3

---

## DL-004 — Transaction state machine

**Date:** 2026-05-22
**Decision:** Ledger transactions follow: Pending → Approved → Settled, with Voided reachable from Pending or Approved. Settled transactions cannot be voided.
**Rationale:** User confirmed this shape. Settled = funds have moved and are reconciled; voiding a settled transaction would require a correcting entry, which is a future concern.
**Captured in PRD:** §6 FR-L2

---

## DL-005 — Standard User visibility

**Date:** 2026-05-22
**Decision:** Standard Users can view all records, including those created by other staff. There is no record-level privacy between staff members.
**Rationale:** User confirmed this explicitly. Operations visibility requires staff to see the full picture, not just their own entries.
**Captured in PRD:** §4 User Roles

---

## DL-006 — No category awareness at Ledger level

**Date:** 2026-05-22
**Decision:** The Ledger does not know about expense categories (Tax, Notary, etc.). Categories are a Processing Tool concern only.
**Rationale:** Keeping the Ledger generic. Other future tools may not use the same categories or any categories at all.
**Captured in PRD:** §6 FR-L1–FR-L5 (no category fields), FR-P3 (categories defined at Processing layer)

---

## DL-007 — Data retention: 10 years

**Date:** 2026-05-22
**Decision:** All transaction records, audit entries, and reports are retained for a minimum of 10 years. No purge of voided or rejected records.
**Rationale:** Business rule stated by user.
**Captured in PRD:** §7 NFRs

---

## DL-008 — Both formatted and raw data export in scope

**Date:** 2026-05-22
**Decision:** Export covers two modes: (1) formatted reports as PDF/Excel, (2) raw transaction data as CSV/Excel. Both are required.
**Rationale:** User confirmed "C. Both" when asked.
**Captured in PRD:** §6 FR-P7

---

## DL-009 — No Landbank API integration

**Rationale:** "Not yet" — deferred to a future release.
